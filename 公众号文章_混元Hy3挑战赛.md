## 一个带娃的痛点，变成了一个产品

我家两个孩子在上小学，每天最头疼的事之一，就是课表。

哪天有体育课要带运动鞋？哪天有美术课要带水彩？书包里该装什么、明天穿什么校服、几点出门不迟到……这些信息散在微信群、纸质课表、老师口头通知里，家长和娃自己都经常搞混。

我平时帮客户做 CRM 实施，算是半个技术人。第一反应是——这事儿做个 App 不就解决了？

但说实话，我不是专业移动开发，Flutter 没碰过，Android 上架那套更是一窍不通。搁以前，这个想法八成烂在收藏夹里。

直到用了 **WorkBuddy（内置腾讯混元 Hy3）**。

两周时间，一个能装到手机上跑的 Android App，从需求到 GitHub 分享页上线，全程就我一个人加一个 AI 搭档。下面把这次经历摊开讲讲。

## 我要的不是一个 Demo，是一个能用的东西

先说目标。我对「课程小管家」的要求挺实在：

- 📅 课表管理：按周看，支持单双周（A/B 周）
- 🎒 书包清单：每天该带什么，提前一晚提醒
- 🔔 本地通知：早上提醒当天课、晚上提醒收拾书包
- 📤 课表导出：图片 / Excel / Word / Markdown，方便发家长群
- 🎨 科目配色：一眼分清语数外
- 📱 真机可装：不是模拟器跑跑，是 APK 装到我手机上

还有一条很关键：**不上应用商店**（软著要花钱办证），走 **GitHub 分享页 + Release 下载**，零成本分发。

## 实战三段式：开发、设计、办公一条龙

### 1️⃣ 软件开发：从"看不懂"到"改得动"

技术栈选了 Flutter + Hive（本地数据库）+ Provider（状态管理）+ GoRouter（路由）。

说实话，最开始 `main.dart` 里那串 Hive 初始化和 `MultiProvider` 我是一脸懵的。但 WorkBuddy 不是只丢一段代码完事——它会给我讲清楚每个 Provider 管什么 Box、为什么这么拆。我改一个课表模型字段，它连带帮我改掉所有引用处，不会出现"编译过了运行崩"的情况。

遇到真机 `INSTALL_FAILED_USER_RESTRICTED`（手机禁 USB 安装）这种坑，它立刻换思路：**adb push 到下载目录 + 手动安装**，绕过去了。

印象最深的是导出功能。我要把课表存成图片发微信，结果多行科目名（比如"少先队活动"）在单元格里底部漏白。Hy3 直接定位到 `Table` 在 `RepaintBoundary` 里缺宽度约束，给了个 `IntrinsicWidth` 方案，一行核心改动搞定。

> 💬 让 AI 写代码时别只说"帮我写个XX"。像我说"手机报 XXX 错，已试过 Y，想绕过去"——上下文越真，方案越准。

---

### 2️⃣ 前端设计：分享页和品牌图，不用找设计师

决定不上应用商店后，需要做两样"门面"：

**A. GitHub 分享页**

要求是响应式、能看截图、能点下载。WorkBuddy 直接产出一套干净的自适应 HTML。我截图命名原来是中文带空格（如 `04 导出分享.png`），导致网页图片裂开——它发现后批量重命名为 `01_schedule.png` 这类 URL 友好的名字，同步修好了 HTML 引用。

**B. 应用市场素材（1080×480 品牌图 + 1080×1920 介绍视频）**

- 品牌图：先用 ImageMagick 合成，结果只出了一半（280×124 的残图），后来改用 **Python PIL** 重做，logo 和文字都做了垂直居中
- 介绍视频：ffmpeg 做 6 张截图淡入淡出，第一次视频编码损坏播不了，改成**先生成独立片段再 concat 合并**的分步法，21 秒竖版 MP4 成功

我一个完全不会 PS/PR 的人，靠对话就把物料搞定了。

---

### 3️⃣ 办公生产：文档和发布，一人顶一个小组

这块最容易被忽略，但也最费时间：

- **使用指南**：出了两个版本——GitHub 公开版（去敏感信息）+ 本地私有版（含 keystore 密码），用 `.gitignore` 规则隔离开
- **开发文档**：783 行，覆盖架构、数据模型、构建发布、排障，直接当 wiki 提仓库
- **版本管理**：简单三文件（pubspec.yaml + app_version.dart + CHANGELOG.md），Hy3 帮我保持一致不漏改
- **GitHub 全流程**：建仓、推送、开 Pages、打 Release 传 APK——命令和 Token 用法一步步列清，我照着敲就行

本来这些要产品+文档+运维三个人干的活，一个下午收工。

## 踩过的坑

真实任务才有真实坑，挑三个最有代表性的：

| 坑 | 现象 | 怎么解决的 |
|---|---|---|
| 密钥库密码泄露 | 一时大意把 `keystore.properties`（含密码）传到了 GitHub | `git rm --cached` 移出跟踪 + 写进 `.gitignore`，还提醒我**历史提交里仍有明文，得换密钥库** |
| 截图尺寸不对 | 手机物理分辨率 1200×2670，应用市场要 1080×1920 | sips 调宽 + convert 裁切，批量规范化 |
| 中文路径裂图 | 网页里 `04 导出分享.png` 加载失败 | 批量重命名为英文，修 HTML 引用 |

最大的收获其实不是修 bug 本身，而是每次都顺手教我"下次怎么不踩"。比如密钥库那次，它没只删文件，而是明确告诉我 Git 历史里还有残留，建议换新密钥库。这种感觉不像在用一个工具，更像跟一个靠谱的搭档干活。

## 给想动手的你三点真心话

**1. 选真实任务，别选练习题。**

我这次就是给自己做个带娃工具，真实需求会逼出真实问题——文章和作品自然就有料。这也正好契合这次挑战赛"完成真实任务"的宗旨。

**2. 把 AI 当搭档，不当搜索引擎。**

多给上下文（环境、已试过的、报错原文），它从"答问机器"变成"一起扛事的队友"。这点差别很大。

**3. 开发设计办公一把抓。**

别只盯写代码——分享页、文档、发布流程这些"周边"往往最耗人。而 Hy3 在三条线上都能顶住，全交出去效率翻倍。
